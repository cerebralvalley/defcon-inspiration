Welcome everyone. My name is Michael Stepankin.&nbsp; In this session we'll dive into Mutual TLS and&nbsp;&nbsp; authentication based on certificates. I'm excited&nbsp; to present some interesting attacks on mTLS,&nbsp;&nbsp; as well as demonstrate vulnerabilities&nbsp; and popular open source software. My&nbsp;&nbsp; purpose is to give you some fresh ideas on&nbsp; how to attack and secure this technology.&nbsp;&nbsp; So what is a Mutual TLS? In short, it's&nbsp; another form of client authentication.&nbsp;&nbsp; Most of you probably know that the browser&nbsp; checks this server certificate. The same idea&nbsp;&nbsp; can be applied to authenticate browsers or TLS&nbsp; clients in general. For example, a server can&nbsp;&nbsp; ask a browser to provide a valid certificate. It&nbsp; can be either used as an alternative to classic&nbsp;&nbsp; password authentication or as an additional&nbsp; factor. This process is defined in the TLS RFC,&nbsp;&nbsp; and based on asymmetric cryptography. During the handshake, the server can send&nbsp;&nbsp; a certificate request message. The client replies&nbsp; with a certificate message that contains a public&nbsp;&nbsp; key. This is followed by a certificate verify&nbsp; message encrypted by private key, so that even if&nbsp;&nbsp; the message is intercepted, a person in the middle&nbsp; would not see any secrets in the plain text.&nbsp;&nbsp; So what exactly is the certificate?&nbsp; Generally speaking, it's a blob of&nbsp;&nbsp; binary encoded data. The most important fields&nbsp; are the Subject, the Issuer, and the Public Key.&nbsp;&nbsp; Subject usually defines a username or&nbsp; host name. And similar to JSON web tokens,&nbsp;&nbsp; the certificate is signed by the Issuer's private&nbsp; key. So the data inside cannot be tampered with. &nbsp; The Issuer field points to another certificate,&nbsp; who signs the certificate. A sequence&nbsp;&nbsp; of certificates sign in each other forms&nbsp; a certificate chain. In order to check the&nbsp;&nbsp; validity of the certificate, the server should&nbsp; build a trusted chain from the client certificate&nbsp;&nbsp; to a trusted root certificate. The client and&nbsp; intermediate certificates are normally sent by&nbsp;&nbsp; the clients, whereas the server only needs&nbsp; to know about trusted root certificates. &nbsp; Let's see how to use mTLS in an example&nbsp; Java application. In the bare minimum,&nbsp;&nbsp; the developers only need to enable it in the&nbsp; application settings and specify the allocation&nbsp;&nbsp; of all trusted root certificates. From the client&nbsp; perspective, such as curl, you need to specify&nbsp;&nbsp; what certificate is sent to the server. The rest&nbsp; of the application code, such as request mappings,&nbsp;&nbsp; is exactly the same as for&nbsp; normal web application. &nbsp; As you can see, this approach brings&nbsp; some advantages and disadvantages&nbsp;&nbsp; compared to traditional authentication&nbsp; methods, such as password. First of all,&nbsp;&nbsp; it might be faster for your application, as&nbsp; the authentication can only happen at the&nbsp;&nbsp; handshake level. So all subsequent HTTP request&nbsp; will be authenticated, saving our CPU time.&nbsp;&nbsp; Also, similar to JSON web tokens, there&nbsp; is a storage benefit, as a server does not&nbsp;&nbsp; need to store all the client certificates,&nbsp; only need to access the root certificate.&nbsp;&nbsp; Speaking of the cons, there's no granular control.&nbsp; As soon as you enable mTLS in your application,&nbsp;&nbsp; all requests have to be authenticated even&nbsp; to the public pages or JSS or CSS resources. &nbsp; Also, any certificate signed by a root certificate&nbsp; can be used for client authentication. Even if&nbsp;&nbsp; this certificate is issued for another purpose,&nbsp; it still can potentially be used for TLS&nbsp;&nbsp; authentication. Furthermore, the Mutual TLS does&nbsp; not specify how the certificate should be issued.&nbsp;&nbsp; It's up to the application to define additional&nbsp; mechanism for certificate issuance. Finally,&nbsp;&nbsp; certificates are always expire and it's&nbsp; just a hassle to rotate them frequently. &nbsp; Before we dive into the attack section,&nbsp; I'm going to briefly mention some of&nbsp;&nbsp; the previous well-known attacks on&nbsp; certificate validation. Obviously,&nbsp;&nbsp; the security of the authentication system&nbsp; depends on the strength of the signature.&nbsp;&nbsp; If you can somehow forge the content of the&nbsp; certificate but keep the same signature, we can&nbsp;&nbsp; completely break the authentication process.&nbsp; Also, since X.509 format is quite complex,&nbsp;&nbsp; just parsing these data structures can lead&nbsp; to buffer and heap overflows. In additional,&nbsp;&nbsp; lack of basic constraint check-in can also be&nbsp; used to bypass the authentication process. &nbsp; I've got to say that in my research, I&nbsp; mostly looked at how mTLS is implemented&nbsp;&nbsp; in applications written in Java.&nbsp; In the Java development kit,&nbsp;&nbsp; most of these attacks are already&nbsp; mitigated on the language level,&nbsp;&nbsp; so I had to take a different approach. At the&nbsp; same time, the attacks I'm going to describe&nbsp;&nbsp; are also applicable to other languages. All right, let's move on to the attack's&nbsp;&nbsp; section. In the first chapter, we'll&nbsp; cover improper certificate extraction.&nbsp;&nbsp; In real life applications, developers often need&nbsp; to access the subject name from the certificate&nbsp;&nbsp; presented during the TLS handshake. For example,&nbsp; they may need it for authorization needs,&nbsp;&nbsp; to understand what is the current username. In&nbsp; Java, you can use sslSession.getPeerCertificates&nbsp;&nbsp; method. Interestingly, this method&nbsp; returns an array of certificates,&nbsp;&nbsp; not a single one. Why? Perhaps because TLS&nbsp; specification defines that clients might send&nbsp;&nbsp; a full chain of certificate, not only one. So I decided to take a look at how different&nbsp;&nbsp; applications use this API. The most common&nbsp; approach I've seen is to take only the first&nbsp;&nbsp; certificate from the array and consider it&nbsp; as a client certificate. This is correct, as&nbsp;&nbsp; mTLS RFC explicitly says that the sender&nbsp; certificate must come first in the list. At&nbsp;&nbsp; the same time, I discovered some rare cases when&nbsp; application disregard this rule, and iterate over&nbsp;&nbsp; the array trying to find a certificate that&nbsp; matches some criteria. It can be dangerous,&nbsp;&nbsp; as the underlying TLS library in Java only&nbsp; verifies the first certificate from the list. &nbsp; One of these examples was of vulnerability I&nbsp; discovered in Keycloak. Keycloak is a popular&nbsp;&nbsp; authorization server that supports OAuth, SAML&nbsp; and other authorization methods, as well as Mutual&nbsp;&nbsp; TLS. Keycloak iterates over all certificates in&nbsp; the array, searching for the one that matches&nbsp;&nbsp; the client_id form parameter. As soon as it&nbsp; found one certificate, it implicitly trusts it,&nbsp;&nbsp; assuming that its signature has already&nbsp; been checked during the TLS handshake. &nbsp; In reality, a client can send as&nbsp; many certificate as they want,&nbsp;&nbsp; so the server only verifies the first one. A&nbsp;&nbsp; potential attacker can exploit this behavior&nbsp; to authenticate under a different username. It&nbsp;&nbsp; is possible to send a list of certificates,&nbsp; where the first one contains one username&nbsp;&nbsp; and properly changed to a root CA. The last&nbsp; certificate might be self signed and belongs&nbsp;&nbsp; to a different user. The client does not even&nbsp; need to provide a valid private key for it. &nbsp; There are a number of endpoints and Keycloak that&nbsp; support mTLS authentication. For exploitation,&nbsp;&nbsp; we need ones that doesn't require any&nbsp; additional factors such as tokens or secrets.&nbsp;&nbsp; Client-management/register-node is a good example,&nbsp; as it mutates user's data. In the first comment,&nbsp;&nbsp; it's shown how we can normally use this&nbsp; API with mTLS. In the second request,&nbsp;&nbsp; we generate a new self signed certificate&nbsp; using Open SSL, and add it to the chain. When&nbsp;&nbsp; we send the second curl request, Keycloak&nbsp; will perform this action on behalf of the&nbsp;&nbsp; user specified in the certificate two,&nbsp; instead of certificate one. Therefore,&nbsp;&nbsp; we can mutate data on the server&nbsp; for any client that supports mTLS. &nbsp; How to fix that? Easy, just use a first&nbsp; certificate from the array. That's exactly&nbsp;&nbsp; how Keycloak patch this vulnerability.&nbsp; It's a good example of how developers&nbsp;&nbsp; provide methods and interfaces that can&nbsp; be misunderstood or used incorrectly.&nbsp;&nbsp; Another common scenario for mTLS deployments&nbsp; is when TLS connection is terminated on the&nbsp;&nbsp; reverse proxy. In this case, the reverse proxy&nbsp; often checks the certificate and forwards it&nbsp;&nbsp; to backend server as an additional&nbsp; header. On this slide, you can see a&nbsp;&nbsp; typical engine configuration to enable mTLS. I've seen a number of systems like that and&nbsp;&nbsp; in most cases, backend service behind nginx do&nbsp; not perform additional validation. Trust in the&nbsp;&nbsp; reverse proxy. This behavior is not exploitable&nbsp; straightaway, but it's not ideal either. Why?&nbsp;&nbsp; Well, first of all, it means that any server&nbsp; in the local network can make request with the&nbsp;&nbsp; header, so the network segment should be properly&nbsp; isolated from the traffic coming from outside.&nbsp;&nbsp; Additionally, if your backend or reverse proxy&nbsp; is affected by a request smuggling vulnerability,&nbsp;&nbsp; its exploitation becomes trivial. Over the&nbsp; past few years, we've seen a lot of requests&nbsp;&nbsp; and header smuggling vulnerabilities, including&nbsp; the latest CVEs in Apache and Nodejs. Be careful&nbsp;&nbsp; when implementing these scenarios, and check the&nbsp; certificate signature on all servers if possible. &nbsp; Moving on to the next chapter,&nbsp; follow the chain: where it leads you?&nbsp;&nbsp; In large systems, servers may not store all&nbsp; root and intermediate certificates locally,&nbsp;&nbsp; but use an external storage instead. RFC&nbsp; defines a concept of a certificate store:&nbsp;&nbsp; an interface you can use to lazily access the&nbsp; certificates during chain validation. These tools&nbsp;&nbsp; are implemented over different protocols&nbsp; such as HTTP, LDAP, FTP or SQL queries. &nbsp; The X.509 format defines some certificate&nbsp; extensions that can contain information where&nbsp;&nbsp; to find the Issuer and CA certificates.&nbsp; For instance, the Authority Information&nbsp;&nbsp; Access extension contains a URL pointing to the&nbsp; Issuer's certificate. If this extension is used&nbsp;&nbsp; for validation, there is a high chance that&nbsp; you can exploit it to perform an SSRF attack.&nbsp;&nbsp; Also Subject, Issuer, Serial and&nbsp; their alternative names can be&nbsp;&nbsp; used to construct SQL and LDAP queries,&nbsp; leaving options for injections attack.&nbsp;&nbsp; When certificates stores are in use, you want&nbsp; to think about these values as insertion points,&nbsp;&nbsp; similar to those ones we have in Burp Suite's&nbsp; Intruder. And what's really good for attackers,&nbsp;&nbsp; all of these values can be used in&nbsp; queries before the signature is checked. &nbsp; To demonstrate an example of this vulnerability,&nbsp; we'll use LDAPCertStore from the Bouncy Castle&nbsp;&nbsp; library. Bouncy Castle is one of the most popular&nbsp; libraries for certificate validation in Java. Here&nbsp;&nbsp; is an example of how you can use this store&nbsp; to build and validate a certificate chain.&nbsp;&nbsp; Under the hood, Bouncy Castles uses a Subject&nbsp; field from the certificate to build an LDAP query.&nbsp;&nbsp; The Subject field is inserted in the filter,&nbsp; and as you can imagine, without any escaping.&nbsp;&nbsp; So if the Subject contains any special characters,&nbsp; it can change the syntax of the LDAP query. &nbsp; In most cases, this can be exploited as&nbsp; a blind LDAP query injection. Therefore,&nbsp;&nbsp; it could be possible to use this vulnerability to&nbsp; extract other fields from the LDAP directory. The&nbsp;&nbsp; exploitation is based on many factors, including&nbsp; whether the application exposes any errors,&nbsp;&nbsp; and it also depends on the&nbsp; structure of the LDAP directory.&nbsp;&nbsp; Anyway, whenever you incorporate user-supplied&nbsp; data into LDAP query, special characters should&nbsp;&nbsp; be properly filtered. That's exactly how this&nbsp; CVE has been patched in the Bouncy Castle code. &nbsp; Next we'll cover revocation. This controversial&nbsp; topic sparked many debates over the years. Similar&nbsp;&nbsp; to JSON web tokens, the beauty of certificate&nbsp; chains is that they can be trusted just based&nbsp;&nbsp; on their signature. But what happens if we need&nbsp; revoke a certificate, so it can no longer be used?&nbsp;&nbsp; The PKIX specification addresses this problem by&nbsp; proposing a special store for revoke certificates,&nbsp;&nbsp; accessible while HTTP or LDAP protocols.&nbsp; Certificate verifiers can use two different&nbsp;&nbsp; methods, OSCP or CRL. Many developers believe&nbsp; that revocation checking is absolutely necessary,&nbsp;&nbsp; whereas others urge to avoid it for performance&nbsp; reasons or use offline revocation lists. &nbsp; Generally speaking, the store location can&nbsp; be hardcoded into the application or taken&nbsp;&nbsp; from the certificate itself. There are&nbsp; two certificate extensions used for that:&nbsp;&nbsp; Authority Information Access&nbsp; and CRL Distribution Points.&nbsp;&nbsp; Looking at it from the hacker's point of view,&nbsp; I think it's incredible that the location of&nbsp;&nbsp; the revocation server can be taken from the&nbsp; certificate. So if application takes URLs&nbsp;&nbsp; relying on extensions to make a revocation&nbsp; check, it can be abused for SSRF attacks. &nbsp; Sadly, for attackers, this normally happens&nbsp; after the signature checks, but in some cases&nbsp;&nbsp; it's still exploitable. Moreover, the LDAP&nbsp; protocol is also supported, at least in Java.&nbsp;&nbsp; Wait, did I hear LDAP in Java? Yes. The&nbsp; standard Java library can use JNDI to&nbsp;&nbsp; access the location locations under some&nbsp; settings. You probably heard that in Java,&nbsp;&nbsp; unmarshaling an LDAP lookup response can lead to&nbsp; a remote code execution. A few years back this&nbsp;&nbsp; problem was reported by Moritz Bechlar, and remote&nbsp; code execution via revocation been patched in JDK.&nbsp;&nbsp; You can check out his blog post for more details. In my research, I decided to check if Bouncy&nbsp;&nbsp; Castle library is also affected. It turned&nbsp; out that the Bouncy Castle can use the CRLDP&nbsp;&nbsp; extension and make calls to an LDAP server.&nbsp; At the same time, Bouncy Castle only fetches&nbsp;&nbsp; a specific attribute from the LDAP response&nbsp; and does not support references. So remote&nbsp;&nbsp; code execution is sadly not possible there.&nbsp; HTTP SSRF is still a viable thing though. &nbsp; I also had a quick look at open source projects&nbsp; that support mTLS and perform revocation checking.&nbsp;&nbsp; One of these projects was Apereo CAS. It's&nbsp; another popular authentication server that&nbsp;&nbsp; is highly configurable. Administrators of&nbsp; Apereo CAS can enable the revocation check&nbsp;&nbsp; using the external LDAP server by specify&nbsp; its address and password in the settings.&nbsp;&nbsp; If the settings are applied, Apereo CAS performs&nbsp; their vocation check for the certificate,&nbsp;&nbsp; fetching the address from the certificates&nbsp; CRLDP extension. I was afraid that this could&nbsp;&nbsp; also to remote code execution, but it turned&nbsp; out that Apereo CAS uses a custom library for&nbsp;&nbsp; LDAP connection, which does not support external&nbsp; codebases or object factories needed for RCE. &nbsp; When I tested this in Apereo CAS, I noticed one&nbsp; interesting behavior. The servers prefers LDAP&nbsp;&nbsp; URLs located inside the certificate, instead of&nbsp; the ones that is configured in the settings. At&nbsp;&nbsp; the same time, Apereo CAS still sends a password&nbsp; from the settings. I quickly set up a testing&nbsp;&nbsp; environment and send a self-signed certificate in&nbsp; the header. This certificate has CRLDP extension&nbsp;&nbsp; with LDAP URL pointing to a netcat listener.&nbsp; After sending this request to Apereo CAS,&nbsp;&nbsp; I managed to receive a request to my netcat&nbsp; listener with a username and password leaked. &nbsp; After reporting this vulnerability, the&nbsp; application developers issued a fix within&nbsp;&nbsp; just one day. They patched it by clearing&nbsp; the login and password from LDAP connection.&nbsp;&nbsp; Therefore, the password leak is no&nbsp; longer possible. But in general, I'd&nbsp;&nbsp; say that using URLs from CRLDP extension is still&nbsp; dangerous, as it broadens the attack surface. &nbsp; To summarize, when you develop an mTLS system&nbsp; or perform a security assessment, I suggest:&nbsp;&nbsp; first, pay attention when extracting&nbsp; usernames from the mTLS chain,&nbsp;&nbsp; as a service only verifies the&nbsp; first certificate in the chain.&nbsp;&nbsp; Second, use Certificate Stores source with&nbsp; tion. It can lead to LDAP and SQL injections.&nbsp;&nbsp; And last but not least, certificate revocation&nbsp; can lead to SSRF or even RCE in the worst case.&nbsp;&nbsp; So it's advice to do the revocation check only&nbsp; after all other checks.
